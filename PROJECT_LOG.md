# Khantoke Gold System — บันทึกโปรเจค

## ภาพรวม

แอประบบจัดการร้านข้าวกล่อง "Khantoke Gold" — ติดตามยอดขาย, บันทึกออเดอร์ลูกค้า, วางแผนการผลิต,
จัดการสต๊อกวัตถุดิบ (รวมน้ำจิ้ม), คำนวณต้นทุน/กำไรต่อเมนู, และจัดการรายชื่อลูกค้า/ผู้จำหน่าย

- **ที่มา:** นำเข้าจากโปรเจค Claude Design (claude.ai/design) ไฟล์ต้นฉบับ `Khantoke Gold System.dc.html`
  (เวอร์ชันแรกพอร์ตเป็นไฟล์ HTML เดี่ยว React+Babel ก่อน แล้วแปลงเป็น PHP+MySQL ตามคำขอวันที่ 2026-07-18)
- **เทคโนโลยี:** PHP (แยกไฟล์ต่อหน้า) + MySQLi + JavaScript (vanilla, ไม่ใช้ framework) —
  ต้องมีเซิร์ฟเวอร์ PHP (เช่น XAMPP/Apache หรือ `php -S`) และฐานข้อมูล MySQL/MariaDB
- **การเก็บข้อมูล:** ฐานข้อมูล MySQL `khantoke_gold` (ผ่าน `db.php`) — ข้อมูลทุกอย่าง (ออเดอร์/สต๊อก/สูตร/
  ลูกค้า/โหมดมืด/วันที่ที่เลือก) real-time ข้ามอุปกรณ์ได้ทันทีที่เข้าฐานข้อมูลเดียวกัน
  หน้าเว็บ poll ข้อมูลใหม่จากเซิร์ฟเวอร์ทุก 5-6 วินาที (ดู `assets/app.js` → `KG.startPolling`)
- **วิธีใช้งาน:**
  1. สร้างฐานข้อมูล: `mysql --default-character-set=utf8mb4 -u root < schema.sql`
     (**ต้องใช้ `--default-character-set=utf8mb4`** ไม่งั้นข้อความภาษาไทยจะเพี้ยนตอนนำเข้า)
  2. ตั้งค่าการเชื่อมต่อใน `db.php` ให้ตรงกับเซิร์ฟเวอร์ MySQL ของคุณ (ค่าเริ่มต้นคือ XAMPP: root ไม่มีรหัสผ่าน)
  3. รันผ่าน Apache (วางโปรเจคใน `htdocs`) หรือทดสอบเร็วๆ ด้วย `php -S localhost:8000` ที่โฟลเดอร์โปรเจค
  4. เปิด `index.php`
- **GitHub:** https://github.com/uthainantang/Khantoke_Gold.git
- **เซิร์ฟเวอร์จริง (VPS):** รันอยู่จริงบน production แล้ว — IP/พอร์ตจริงเก็บไว้ใน `DEPLOY_PRIVATE.md`
  (ไฟล์ local เท่านั้น ไม่ commit ขึ้น git เพราะ repo นี้เป็น public)

## การติดตั้งบน VPS

รันอยู่บน VPS เดียวกับโปรเจคอื่น (iqbot, lottery-analytics) โดยตั้งใจแยกให้ไม่ชนกับของเดิมทุกจุด
(ดู URL จริงได้ใน `DEPLOY_PRIVATE.md`):

- **ตำแหน่งไฟล์:** `/var/www/khantoke-gold` (เจ้าของไฟล์ `www-data`)
- **เว็บเซิร์ฟเวอร์:** nginx site ใหม่ `/etc/nginx/sites-available/khantoke-gold` → PHP-FPM ผ่าน
  `unix:/run/php/php8.3-fpm.sock` — **listen port 8090** (พอร์ต 80/443/8080/8551/3000 ถูกโปรเจคเดิมใช้อยู่แล้ว
  จึงเลี่ยงมาใช้พอร์ตนี้แทน ไม่ได้แก้ config เดิมของ iqbot/lottery-analytics เลย)
- **ที่ติดตั้งเพิ่มบน VPS** (ของเดิมไม่มี PHP/MySQL มาก่อน): `php-fpm` + `php-mysql` (PHP 8.3.6),
  `mariadb-server` + `mariadb-client` (MariaDB 10.11) — ติดตั้งผ่าน `apt`, เปิดใช้งานเป็น systemd service
- **ฐานข้อมูล:** สร้างฐานข้อมูล `khantoke_gold` แยกต่างหาก พร้อม MySQL user เฉพาะของแอปนี้
  (`khantoke_app`, สิทธิ์จำกัดเฉพาะฐานข้อมูลนี้ — **ไม่ใช้ root**) รหัสผ่านสุ่มถูกตั้งไว้ใน `db.php`
  **บน VPS เท่านั้น** (ไฟล์ `db.php` ใน Git ยังเป็นค่า default สำหรับ local dev — ไม่ commit รหัสผ่าน production ขึ้น Git)
- **ข้อมูลเริ่มต้น:** ล้างข้อมูลตัวอย่างออกหมดแล้ว (ตามที่ขอไว้ก่อนหน้านี้) เหลือแต่โครงสร้างตาราง พร้อมให้กรอกข้อมูลจริง
- **การอัปเดตในอนาคต:** อัปโหลดไฟล์ใหม่ทับที่ `/var/www/khantoke-gold` แล้ว reload PHP-FPM ถ้าจำเป็น
  (ไม่ต้องแตะ `db.php` บนเซิร์ฟเวอร์อีกถ้าไม่ได้เปลี่ยนโครงสร้างฐานข้อมูล)

## โครงสร้างไฟล์

- `db.php` — ไฟล์เชื่อมต่อฐานข้อมูล (MySQLi) ไฟล์เดียว ใช้ร่วมกันทุกหน้า/ทุก API
- `schema.sql` — สร้างฐานข้อมูล ตาราง และข้อมูลตัวอย่างเริ่มต้น
- `includes/functions.php` — ฟังก์ชันช่วยเหลือร่วม (แปลงวันที่ไทย, format เงิน, สถานะสต๊อก, JSON helpers)
- `includes/header.php`, `includes/footer.php` — ส่วนหัว/ท้ายหน้าที่ใช้ร่วมกันทุกหน้า (นำทาง, toast, ฯลฯ)
- `index.php`, `orders.php`, `production.php`, `stock.php`, `recipes.php`, `settings.php` — หน้าเพจแยกไฟล์ 6 หน้า
- `api/*.php` — API endpoints (dashboard, orders, production, stock, recipes, vendors, settings)
  รับ-ส่งข้อมูลเป็น JSON อ่าน/เขียนฐานข้อมูลผ่าน MySQLi
- `assets/style.css`, `assets/app.js` — สไตล์และ JS ที่ใช้ร่วมกันทุกหน้า (polling, toast, สลับโหมดมืด)
- `assets/dashboard.js`, `orders.js`, `production.js`, `stock.js`, `recipes.js`, `settings.js` — JS เฉพาะแต่ละหน้า
- `PROJECT_LOG.md` — ไฟล์นี้ บันทึกความเป็นมาและประวัติการแก้ไข

## หน้าจอในแอป

1. **หน้าหลัก (Dashboard)** — ยอดขาย/กำไร/ต้นทุนวันนี้, แจ้งเตือนสต๊อกต่ำ, เมนูขายดี, รายการออเดอร์ + ปุ่มจ่ายเงิน
2. **ออเดอร์ (Orders)** — เลือกวันที่, เลือกเมนูที่ทำวันนี้, รับออเดอร์ต่อลูกค้าแบบเพิ่ม/ลดจำนวน
3. **ผลิต (Production)** — บันทึกปริมาณวัตถุดิบที่ใช้ 3 สถานี (น้ำซุป/หุงข้าว/เนื้อ) พร้อมหักสต๊อกอัตโนมัติ
4. **สต๊อก (Stock)** — จัดการวัตถุดิบ (เพิ่ม/แก้ไข/ลบ) + จัดการน้ำจิ้ม (ส่วนผสมต่อรอบ, ทำรอบใหม่)
5. **เมนู (Recipes)** — จัดการเมนู, ส่วนผสมต่อกล่อง, คำนวณต้นทุน/กำไรอัตโนมัติ
6. **ตั้งค่า (Settings)** — จัดการรายชื่อลูกค้า/ผู้จำหน่าย

    `substr(...) === ...` ให้ใช้ได้กับ PHP 7.4 ขึ้นไป
- ทดสอบทุก endpoint (GET/POST ทุก action ของทุกโมดูล) ผ่าน XAMPP MySQL จริง — ไม่มี PHP fatal error
- ทดสอบ flow จริงผ่านเบราว์เซอร์: เพิ่มออเดอร์ → หักสต๊อกจากหน้าผลิต → แก้ราคาสต๊อก → เพิ่มวัตถุดิบเมนู
  พร้อม preview ต้นทุน real-time → สลับโหมดมืด (บันทึกลง MySQL ข้ามอุปกรณ์ได้จริง ไม่ใช่แค่ localStorage)
- รีเซ็ตฐานข้อมูลกลับเป็นข้อมูลตัวอย่างเริ่มต้นหลังทดสอบเสร็จ

### 2026-07-18 — ปรับให้ใช้งานได้ดีทั้งคอมพิวเตอร์และมือถือ (responsive)
- เดิมแอปถูกล็อกความกว้างไว้ที่ 430px เสมอ (ดีไซน์ต้นฉบับเป็นมือถือ) — บนจอคอมพิวเตอร์จะเห็นเป็นแถบแคบ
  ลอยกลางจอ ใช้งานได้แต่ไม่ได้ใช้พื้นที่จอให้เป็นประโยชน์
- เพิ่ม CSS media query (`min-width: 900px`) ใน `assets/style.css`: จอกว้างจะขยายเต็มความกว้าง, เมนูนำทาง
  ย้ายจากแถบล่าง (มือถือ) ไปเป็นแถบข้างซ้าย (คอมพิวเตอร์), เนื้อหาจำกัดความกว้างไว้ที่ ~760px ให้อ่านง่าย
  ไม่ยืดเต็มจอจนดูโหว่
- การ์ดที่เป็นลิสต์ยาว (เช่น การ์ดวัตถุดิบในหน้าสต๊อก) ใช้ `grid-template-columns: repeat(auto-fit, minmax(...))`
  แทนการล็อก 2 คอลัมน์ตายตัว — มือถือยังเห็น 2 คอลัมน์เหมือนเดิม แต่จอกว้างขยายเป็น 3-4 คอลัมน์อัตโนมัติ
  ส่วนแถวฟอร์ม/KPI ที่มีแค่ 2 ช่องจะไม่ขยายเกิน 2 คอลัมน์ (เพราะมีแค่ 2 รายการให้วาง)
- แก้ `includes/header.php` เพิ่ม wrapper `.kg-main` ครอบ header+หัวข้อหน้า+เนื้อหา เพื่อให้ CSS จัดตำแหน่ง
  เทียบกับแถบข้างซ้ายได้ — ไม่ต้องแก้ไฟล์ JS ของแต่ละหน้าเลย (การ์ด/กริดใช้ class เดิมอยู่แล้ว)
- ทดสอบผ่าน VPS จริงด้วยการวัด layout metrics (ขนาด/ตำแหน่งจริงของ element): ยืนยันจอกว้าง 1440px
  ใช้แถบข้างซ้าย 220px + เนื้อหากว้าง 760px + กริดวัตถุดิบขยายเป็น 4 คอลัมน์, จอมือถือ 375px ยังคงแถบล่าง
  + กริด 2 คอลัมน์เหมือนเดิมทุกประการ — ไม่มีอะไรเปลี่ยนสำหรับผู้ใช้มือถือ

- **แก้ที่ `assets/orders.js` และ `assets/expenses.js`:**
  - เพิ่ม `width:100%` ให้ `<table>` — คอลัมน์ที่ตั้ง `table-layout:fixed` ไว้จะขยายตามสัดส่วนเดิมจนเต็มพื้นที่
    การ์ดจริงเสมอ ไม่เหลือช่องว่างอีกต่อไป (คอลัมน์เมนูกว้างขึ้นเป็นผลพลอยได้ด้วย ช่วยให้ตัดคำสวยขึ้น)
  - เปลี่ยนจาก `overflow-wrap:break-word` เป็น `overflow-wrap:anywhere` (ไม่ใช้ `word-break:break-word`
    อีกต่อไป) — ให้เบราว์เซอร์พยายามตัดตรงจุดที่เหมาะสมตามปกติก่อน (ระหว่างพยางค์ไทย/ก่อนตัวเลข) แล้วค่อย
    บังคับตัดเป็นทางเลือกสุดท้ายถ้าไม่มีที่ให้ตัดจริงๆ เท่านั้น
- ทดสอบยืนยันบน VPS จริงที่ขนาดจอ 412px (ใกล้เคียงมือถือ Android จริงมากกว่า 375px ที่ทดสอบไปก่อนหน้า):
  ตารางกว้างเต็ม 382px เท่ากับพื้นที่การ์ดพอดี (ไม่มีช่องว่างอีกแล้ว), หัวคอลัมน์เมนูเหลือแค่ 2 บรรทัด
  (จากเดิม 4 บรรทัด) และตรวจสอบด้วย `Range.getClientRects()` ยืนยันว่าตัวเลข "20" ไม่ถูกตัดขาดจากกันแล้ว
